# Design Context — autotel-devtools widget

## Users
Developers and SREs debugging OpenTelemetry traces locally. The UI is an
embedded custom-element widget (`<autotel-devtools>`, Shadow DOM) injected into
the developer's own running app, used during active debugging — scanning trace
lists, reading waterfalls, chasing errors. High information density, fast
keyboard-driven scanning. Used in both bright and dark environments, so the
product ships light / dark / system themes.

## Brand Personality
Fast, dense, precise. A calm professional instrument — closer to an
oscilloscope or a pro audio tool than a consumer dashboard. It should feel
quiet and trustworthy, never flashy. Confidence through legibility and rhythm,
not decoration.

## Aesthetic Direction
- **Typography duality** — refined UI sans (**Hanken Grotesk**) for all chrome
  (tabs, headings, labels, buttons, prose); **JetBrains Mono** reserved strictly
  for data (trace/span IDs, durations, attribute keys/values, code, log bodies).
  This kills the "everything is monospace = technical" tell.
- **Color** — OKLCH token system, neutrals subtly tinted toward the brand hue
  (cool blue, ~255°). No pure black/white. Blue accent, used sparingly (~10%).
- **Spacing** — 4pt scale (Tailwind's default base), varied for hierarchy.
- **Motion** — restrained; reserved for state changes (modal/panel entrances).
  Always gated behind `prefers-reduced-motion`.
- Anti-references: neon-on-black "AI dashboard," glassmorphism, gradient text,
  side-stripe accent borders, mono-everything.

## Design Principles
1. **Legibility first.** It's a debugging instrument — clarity beats character.
2. **Data is monospace, chrome is sans.** Never blur the two.
3. **Accent is rare.** Blue earns attention because it's scarce.
4. **Tokens, not one-offs.** Refine the central `--at-*` / semantic tokens so
   changes propagate; avoid per-component color/spacing drift.
5. **Calm motion.** Motion confirms a state change or it doesn't happen.
